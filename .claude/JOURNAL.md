5. **Task [Extended] - mmBERT INT8 SmoothQuant notebook** (v0.1.0): Built notebook `00-kj-mmbert-int8-smoothquant.ipynb` quantizing any mmBERT checkpoint to INT8 with SmoothQuant and measuring quality<br>
   **Result**: Stage-0 model-prep notebook for the statement encoder. Post-training-quantizes `jhu-clsp/mmBERT-base` (multilingual ModernBERT, 307M params - the large vocab embedding dominates the 149M backbone) to W8A8 INT8, measuring how far INT8 perturbs the embedding geometry. Implemented SmoothQuant inline: forward-pre hooks collect per-input-channel activation maxima on a disjoint calibration corpus, the smoothing scale `s = act^alpha / weight^(1-alpha)` (alpha 0.5) folds into the preceding `LayerNorm` (divide) and the `Linear` input columns (multiply) - identity in FP, but it flattens the activation outliers that wreck naive INT8. INT8 is fake-quantized (per-channel symmetric weights, per-tensor dynamic activations) via a `QuantLinear` wrapper so quality is measured without INT8 kernels. Quality = Pearson/Spearman of pairwise cosine similarities vs FP32 over a built-in 26-sentence eval corpus, plus a naive-INT8 ablation: SmoothQuant 0.924 Pearson vs naive 0.840 (+0.083), 43 LayerNorm→Linear folds (layer-0 attn_norm is Identity, skipped). Per the user's follow-up, generalized to any mmBERT (original or fine-tuned, Hub id or local path) via `AutoModel` encoder extraction, a `model_type=="modernbert"` guard, FP32 load, and try/except folds. ModernBERT caveats handled: `reference_compile=False` + `attn_implementation="eager"` avoid the first-forward `torch.compile` hang; ran on the RTX 5000 Ada (idx 2). Verified end-to-end via nbconvert (logged to `logs/00-mmbert-int8-smoothquant.log`, 0 errors); wrote `reports/00-mmbert-int8-smoothquant-metrics.json` and `models/00-mmbert-smoothquant-int8.pt`.

6. **Task [Extended] - Improve mmBERT quantization + real OpenVINO INT8** (v0.1.0): Reworked notebook `00` from a per-tensor fake-quant study into an evidence-driven quantization study plus a deployable OpenVINO INT8 artifact (planned + approved via plan mode)<br>
   **Result**: Part A (PyTorch/GPU simulation) added per-token dynamic activation quant alongside per-tensor, a SmoothQuant alpha sweep, a per-component ablation, and in-domain evaluation on notebook 01's 163 statements (`data/interim/01-statements.parquet`, with synthetic fallback). The decisive finding: per-token activation quant is the dominant lever - naive per-tensor 0.929 Pearson vs naive per-token 0.998; SmoothQuant adds only ~0.001 once activations are per-token (per-row scaling already tames the outliers SmoothQuant targets). Best alpha 0.6 (sweep flat ~0.998), all four projections ~0.999 solo so mixed precision was not triggered; chosen config (alpha 0.6, per-token, full INT8) reaches in-domain Pearson 0.9987 vs the prior notebook's 0.924. Part B (OpenVINO/CPU) deploys for real: `optimum-intel`'s OV exporter rejects ModernBERT, so used `openvino.convert_model` directly on the torch module (architecture-agnostic graph trace) then `nncf.quantize(model_type=TRANSFORMER, AdvancedSmoothQuantParameters(matmul=best_alpha))`. The deployed INT8 reaches Pearson 0.956 / Spearman 0.983 / emb-cos 0.992 vs FP32, 1.41x faster (29.1→20.6 ms/sentence CPU) and 1.98x smaller (615→310 MB), saved to `models/00-mmbert-openvino-int8/`. Honest gap surfaced: the simulation's 0.999 is an upper bound from per-token dynamic activations, but NNCF's OV INT8 runtime uses per-tensor static activation quant, so real fidelity lands at 0.956 - reaching the simulated peak in production needs a per-token-dynamic INT8 runtime (ONNX Runtime / torch.ao), not OpenVINO default. De-risked both hard steps in throwaway scripts before baking into the notebook. Side effect: installing `optimum-intel[openvino]` (openvino 2026.2.1, nncf 3.2.0) downgraded transformers 5.10.1 → 5.0.0 env-wide; mmBERT still loads, verified. 7. **Task [Extended] - GPU quantization (torchao FP8), notebook merge, throughput saturation** (v0.1.0): Added a GPU-targeted quantization path and consolidated the quantization work into one notebook plus a saturation sweep (plan-mode approved)<br>
   **Result**: OpenVINO is CPU-only on NVIDIA, so the GPU target needed a different toolchain. Swept torchao schemes on the RTX 5000 Ada: the decisive blocker was that torchao quant is only fast once `torch.compile` fuses the kernels, and `torch.compile` needs a C compiler the env lacked - installed conda-forge `c-compiler`/`cxx-compiler` and pointed `CC`/`CXX` at it. Winner is **FP8 dynamic e4m3 + compile**: 2.16x over bf16-eager, 1.15x over compiled bf16, 0.999 fidelity; INT8-dynamic was slower than compiled bf16 and INT8-weight-only catastrophic (0.03x); compiling bf16 alone already gives 1.85x. Merged the SmoothQuant study + OpenVINO CPU deploy (old nb `00`) with the GPU sweep into one `notebooks/02-kj-mmbert-quantization.ipynb` (47 cells, Part A study / Part B CPU OV / Part C GPU, emitting the best model per target); removed nb `00`/`02`-standalone. Added `notebooks/03-kj-mmbert-throughput-saturation.ipynb`: the throughput knee is ~batch 32 (L2-cache-resident), peak ~3000-3180 sent/s settling to ~2100, FP8 only ~2-11% over compiled bf16 even saturated, ~15-26x headroom from batch 1. CPU per-core via OpenVINO async: THROUGHPUT hint, 16 streams, batch 64 → 195 sent/s (6.1 sent/s/core) is the CPU optimum. Wrote `docs/mmbert-quantization-solution.md` (two-target solution, per-token-vs-per-tensor, the compiler gate, an Inference section with recommended optima and per-core numbers). Shipped `stellars/mmBERT-base-openvino-int8` to HF (public, model card via the `my-huggingface` skill, vault token, public parity Pearson 0.979). Honest conclusion documented: quantization is a memory/throughput lever, not a single-request latency lever (CPU INT8 1.4x, GPU FP8 modest) - batching plus compile dominate, and the model is ~⅓ matmul so Amdahl caps INT8's bit-width benefit.
